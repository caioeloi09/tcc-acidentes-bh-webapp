# Data Model & Migration Pipeline

This document describes the runtime database schema and the pipeline that
produces it from the BHTrans open data.

## Source data

The raw data comes from the BHTrans open-data portal and is cleaned and
consolidated in **phase 1** of the project into a single wide CSV,
`data/dataset_consolidado.csv`. That CSV joins four original BHTrans tables:

| Original table | Approx. rows |
|----------------|-------------:|
| Accidents      | 97,498 |
| Locations      | 126,879 |
| Vehicles       | 177,705 |
| People         | 210,977 |

Each row in the consolidated CSV is at the *person/vehicle* grain (one accident
appears in multiple rows). The migration script collapses it to **one row per
accident**.

## Migration pipeline — `scripts/migrate_data.py`

Run with:

```bash
python3 scripts/migrate_data.py \
  --input  data/dataset_consolidado.csv \
  --output backend/data/accidents.db
```

(Defaults resolve to those same paths relative to the script.) The script runs in
five stages:

1. **Read & clean** — load the CSV with pandas and strip whitespace from string
   columns.
2. **Parse dates** — parse `DATA_HORA_BOLETIM` (`DD/MM/YYYY HH:MM`) and derive
   `year`, `month`, `hour`, and `weekday` (`0=Monday`).
3. **Aggregate to one row per accident** — group by `boletim`, taking the first
   value for accident-level attributes and computing:
   - `total_people` = distinct people involved (`Nº_envolvido`)
   - `total_fatalities` = count of rows whose `desc_severidade == "FATAL"`
   - `total_vehicles` = distinct vehicles (`seq_veic`)
4. **Convert coordinates** — transform `COORDENADA_X` / `COORDENADA_Y` from UTM
   (SIRGAS 2000, zone 23S) to WGS84 `latitude`/`longitude` with the built-in
   `utm_to_latlon()` function; invalid/zero coordinates are dropped. See
   [ADR-0004](./architecture/decisions/2026-03-31-utm-to-wgs84-conversion.md).
5. **Write SQLite** — replace the `accidents` table, then create indexes.

Typical output: **97,498** accidents written, **~88,437** with valid coordinates,
and **882** flagged fatal (`is_fatal = 1`).

> **Note on fatality counts.** `is_fatal` is the per-accident fatality flag
> derived from `INDICADOR_FATALIDADE` (~882 accidents), while `total_fatalities`
> sums the number of fatal victims per accident. Use whichever is appropriate and
> keep the distinction consistent across the report and the dashboard.

## Schema — `accidents` table

One table, generated by the migration script (`ddl-auto: none` on the backend —
Hibernate does **not** own the schema).

| Column | SQLite type | Meaning |
|--------|-------------|---------|
| `id` | INTEGER (PK) | Surrogate key (row index) |
| `boletim` | TEXT | Police-report number (unique per accident) |
| `date_time` | TEXT | ISO-8601 timestamp |
| `year` | INTEGER | Year of the accident |
| `month` | INTEGER | Month (1–12) |
| `hour` | INTEGER | Hour of day (0–23) |
| `weekday` | INTEGER | 0=Monday … 6=Sunday |
| `accident_type` | TEXT | `DESC_TIPO_ACIDENTE` |
| `district` | TEXT | Regional (`DESC_REGIONAL`) |
| `neighborhood` | TEXT | Neighborhood (`nome_bairro`) |
| `street_type` | TEXT | Roadway type (`tipo_logradouro`) |
| `street_name` | TEXT | Roadway name (`nome_logradouro`) |
| `weather` | TEXT | Weather (`DESC_TEMPO`) |
| `pavement` | TEXT | Pavement (`PAVIMENTO`) |
| `is_signposted` | INTEGER | 1 if `LOCAL_SINALIZADO == "SIM"`, else 0 |
| `speed_limit` | INTEGER | Posted speed limit (`VELOCIDADE_PERMITIDA`) |
| `is_fatal` | INTEGER | 1 if `INDICADOR_FATALIDADE == "SIM"`, else 0 |
| `ups_value` | INTEGER | UPS severity value (`VALOR_UPS`) |
| `ups_description` | TEXT | UPS severity label (`DESCRIÇÃO_UPS`) |
| `total_people` | INTEGER | Distinct people involved |
| `total_fatalities` | INTEGER | Fatal victims in the accident |
| `total_vehicles` | INTEGER | Distinct vehicles involved |
| `latitude` | REAL | WGS84 latitude (nullable) |
| `longitude` | REAL | WGS84 longitude (nullable) |

### Indexes

Created at the end of the migration to support the aggregation/map queries:

- `idx_accidents_district (district)`
- `idx_accidents_neighborhood (neighborhood)`
- `idx_accidents_year_month (year, month)`
- `idx_accidents_type (accident_type)`
- `idx_accidents_coords (latitude, longitude)`

## Consumers

- **Backend** — maps this table to the `Accident` JPA entity
  (`@Table(name = "accidents")`) via Spring Data JPA / the
  `hibernate-community-dialects` SQLite dialect.
- **ML service** — reads the same file directly with `sqlite3`
  (`SELECT * FROM accidents`), using `hour`, `weekday`, `year`, `speed_limit`,
  `is_signposted`, `is_fatal`, `month`, `district`, and `accident_type`.

## Regenerating the database

The database is a **build artifact**, not source-controlled data of record. To
rebuild it from scratch, obtain `dataset_consolidado.csv` (see the root README)
and re-run the migration script. Both the backend and the ML service then pick up
the new `backend/data/accidents.db`.
